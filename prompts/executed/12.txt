Implement Leira v1.1: Worker Registry

Mission

Extend Leira from a machine that executes one explicitly supplied worker into a machine that can mechanically resolve named workers.

The goal is not routing.

The goal is not intelligence.

The goal is not automatic worker selection.

The goal is to prove that workers can be registered, looked up, and dispatched by explicit name without changing dispatcher logic.

The machine must be able to say:

> A worker named "EchoWorker" was registered.
>
> Here is the worker associated with that exact name.
>
> The worker was found or not found.
>
> No reasoning occurred.
>
> No worker was chosen automatically.

Nothing more.

Typed Result

```python
@dataclass(frozen=True)
class RegisterResult:
    success: bool
    worker_name: str | None
    error_type: str | None = None
```

Worker Name Rules

Worker names:

* come from worker.name
* must be non-empty strings
* must be unique
* are case-sensitive
* are immutable after registration
* are provenance metadata
* are not routing metadata

Duplicate registration returns:

```python
error_type="DUPLICATE_WORKER"
```

Invalid worker returns:

```python
error_type="INVALID_WORKER"
```

Unknown lookup returns:

```python
None
```

Case Sensitivity

If registered:

```python
echoworker
```

then lookup:

```python
EchoWorker
```

must fail.

Honesty requires exact names.

Ledger Integration

Successful registration must append:

```text
worker_registered
```

Rejected registration must append:

```text
worker_registration_rejected
```

Registration event content must include:

```python
worker_name
status
error_type
```

Successful artifact:

```python
{
    "type": "worker_registration",
    "content": {
        "worker_name": "EchoWorker",
        "status": "REGISTERED",
        "error_type": None
    }
}
```

Rejected artifact:

```python
{
    "type": "worker_registration",
    "content": {
        "worker_name": "EchoWorker",
        "status": "REJECTED",
        "error_type": "DUPLICATE_WORKER"
    }
}
```

Registry-Ledger Atomicity

Do not allow registry memory and ledger history to diverge.

For successful registration:

1. validate worker
2. append worker_registered to ledger
3. update in-memory registry only after ledger append succeeds

If ledger append fails:

* do not store worker in registry
* return typed failure
* error_type="STORAGE_FAILURE"

For rejected registration:

* append worker_registration_rejected where possible
* return typed failure
* do not store worker in registry

No successful in-memory registration may exist without a ledger event.

Dispatcher Integration

Add:

```python
dispatch_by_name(
    intent_id: str,
    worker_name: str
)
```

Algorithm:

1. lookup worker by exact name
2. if worker missing, return:

```python
UNKNOWN_WORKER
```

3. otherwise delegate directly to:

```python
dispatch_once(
    intent_id,
    worker
)
```

No extra logic.

No special path.

No fallback.

No routing.

No worker choice.

Projection

Create:

```text
worker_projection
```

Fields:

```python

```

Projection is derived.

Projection loss is recoverable.

registered_at must come from the ledger event timestamp.

Do not use datetime.now() during rebuild.

Projection Rebuild

Extend rebuild logic to reconstruct:

```text
worker_projection
```

from ledger events.

Only:

```text
worker_registered
```

events create registered workers in projection.

Rejected registration events do not create registered workers.

Rebuild must be deterministic.

Rebuild must be idempotent.

Rebuild must be transactional.

Audit Extension

Extend audit to verify:

* worker names are unique
* registration events exist
* duplicate registrations are reported
* worker_projection matches ledger
* registered_at equals ledger event created_at
* last_event_id equals worker_registered event id
* rejected registrations do not appear as registered workers
* registry introduces no routing semantics

Audit remains read-only.

Audit reports disagreement.

It does not repair.

Registry vs Projection

The in-memory registry cannot be reconstructed from the ledger because worker objects are process references.

The worker_projection can be reconstructed from the ledger because names are history.

Do not pretend ledger replay can recreate Python objects.

It can only recreate the record that a name was registered.
