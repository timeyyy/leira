Implement Leira v1.2: Claim Store

Mission

Extend Leira from a machine that can perform work into a machine that guarantees exclusive access to work.

The goal is not scheduling.

The goal is not parallelism.

The goal is not load balancing.

The goal is to prove that an intent can be claimed mechanically by exactly one owner before execution.

The machine must be able to say:

> This intent was unclaimed.
>
> It was claimed by owner X.
>
> No other owner may claim it.
>
> Here is who owns it.
>
> Here is when it was claimed.
>
> If ownership is orphaned, it remains visible.

Nothing more.

Claim Ownership

Create:

```text
intent_claim_projection
```

Fields:

```python
intent_id
claim_id

```

Rules

Only one active claim may exist per intent.

A claim belongs to exactly one owner.

Owners are opaque strings.

Examples:

```python
dispatcher-1
worker-a
process-17
```

The kernel does not interpret owner identity.

@dataclass(frozen=True)

Validation Rules

Claim requires intent status:

```text
PENDING
```

Claim fails if:

* intent is missing
* owner_id is empty
* owner_id is not a string
* intent status is not PENDING
* intent is already actively claimed

Suggested errors:

```python
UNKNOWN_INTENT
INVALID_OWNER
INVALID_STATUS
ALREADY_CLAIMED
STORAGE_FAILURE
```

Release Rules

Release requires:

* active claim exists
* owner_id matches active claim owner

Release fails if:

* intent has no active claim
* owner_id does not match current owner
* owner_id is invalid

Suggested errors:

```python
NO_ACTIVE_CLAIM
OWNER_MISMATCH
INVALID_OWNER
STORAGE_FAILURE
```

Projection

Create:

```text
intent_claim_projection
```

Fields:

```python
intent_id
claim_id
owner_id
claimed_at
last_event_id
```

Projection is derived.

claimed_at must come from the ledger event timestamp.

Never use:

```python
datetime.now()
```

during rebuild.

Projection Rebuild

Extend rebuild logic.

Scan events chronologically:

```text
intent_claimed
intent_released
```

Reconstruct active claims.

Rules:

* intent_claimed creates active claim.
* intent_released removes active claim only if owner_id and claim_id match.
* released claims disappear from projection.
* a claim without release remains active.
* orphaned claims remain visible.
* duplicate active claims must be detectable by audit.

Rebuild must be:

* deterministic
* idempotent
* transactional

Do not silently overwrite duplicate active claims.

Do not repair malformed history.

Audit Extension

Verify:

* at most one active claim exists per intent
* claim owner matches ledger
* claim_id matches ledger
* claimed_at equals ledger event created_at
* last_event_id equals claim event id
* duplicate active claims are detected
* released claims disappear from projection
* release owner matches claim owner
* release claim_id matches active claim_id
* claim projection matches history
* orphaned claims are visible, not repaired

Audit remains read-only.

Audit reports disagreement.

It does not repair.

Dispatcher Integration

Before execution, dispatcher must claim:

```python
claim_intent(intent_id, owner_id)
```

Dispatcher executes only if claim succeeds.

After terminal state, dispatcher must release:

```python
release_claim(intent_id, owner_id)
```

If release fails after execution:

* record typed failure
* do not pretend claim was released
* leave orphaned claim visible
* audit should surface it

No automatic retry.

No orphan cleanup.

Keep dispatcher dumb.

Preferred wrapper:

```python
dispatch_and_track(
    intent_id,
    owner_id,
    worker
)
```

Algorithm:

1. claim intent
2. if claim fails, return typed DispatchResult with claim error
3. dispatch once
4. release claim
5. return dispatch result, preserving release failure if relevant

Failure Model

Claim rejection is data.

Release rejection is data.

Orphaned claim is visible state.

Not automatic cleanup.

A crashed process may leave an active claim forever.

This is intentional.

The honest machine exposes ownership problems.

It does not guess.

Definition of Done

Leira can establish exclusive ownership without establishing intelligence.

One intent.

One active claim.

One owner.

No duplicate execution.

Orphans are visible.

Claims coordinate.

They do not decide.
