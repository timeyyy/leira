Implement Leira v1.3: Receipt Bundles

Mission

Extend Leira from a machine that records events into a machine that can package evidence.

The goal is not reporting.

The goal is not summarization.

The goal is not explanation.

The goal is to prove that every event associated with one intent can be exposed as a deterministic, complete receipt bundle.

The machine must be able to say:

> This intent existed.
>
> It was claimed or rejected.
>
> A worker executed or failed to execute.
>
> Artifacts were written.
>
> The intent completed, failed, or remains in progress.
>
> Here are the receipts that prove it.

Nothing more.

Receipt Bundle

Use the existing LedgerEvent dataclass from the ledger module.

Implement:

```python
@dataclass(frozen=True)
class ReceiptBundle:
    intent_id: str
    first_event_id: str
    last_event_id: str
    event_count: int
    events: list[LedgerEvent]
```

Rules

Receipt bundles are views.

Receipt bundles contain ledger events.

Receipt bundles do not own truth.

Ledger remains authoritative.

Receipt API

Implement:

```python
get_receipt_bundle(
    intent_id: str
) -> ReceiptBundle | None
```

Implement:

```python
list_receipt_events(
    intent_id: str
) -> list[LedgerEvent]
```

Implement:

```python
export_receipt_bundle(
    intent_id: str
) -> dict
```

Source of Truth Rule

`list_receipt_events` must query the ledger directly by:

```python
intent_id
```

Do not read events from receipt_projection.

The projection may be used for verification.

It must not be the source of bundle events.

Chronology Rules

Events must appear in ledger order.

Never sort receipt events by timestamp.

Use ledger ordering only.

History outranks clocks.

Timestamps may collide.

Ledger order is truth.

Receipt Scope

Receipt bundle includes all ledger events sharing:

```python
intent_id
```

Examples:

```text
intent_submitted
intent_rejected
intent_claimed
intent_claim_rejected
state_running
artifact_written
intent_completed
intent_failed
intent_released
intent_release_rejected
```

Rejected events are included.

Failure events are included.

In-progress events are included.

Nothing is hidden.

Bundle Invariants

For a bundle:

```python
first_event_id
```

must equal the first event in ledger order.

```python
last_event_id
```

must equal the last event in ledger order.

```python
event_count
```

must equal:

```python
len(events)
```

If an intent is still in progress:

* bundle still exists
* last_event_id is the most recent event
* no terminal state is required

If an intent is completed or failed:

* last_event_id should correspond to the most recent event after terminal handling
* audit should detect illegal events after terminal state

Export Determinism

Export format must be deterministic.

Use canonical JSON serialization:

```python
json.dumps(
    export,
    sort_keys=True,
    separators=(",", ":")
)
```

Exporting the same bundle twice must produce byte-identical JSON.

Receipt Projection

Create:

```text
receipt_projection
```

Fields:

```python
intent_id
first_event_id
last_event_id
event_count
updated_at
```

Projection is derived.

`updated_at` must come from the timestamp of:

```python
last_event_id
```

Never use:

```python
datetime.now()
```

during rebuild.

Projection Rebuild

Extend rebuild logic to reconstruct:

```text
receipt_projection
```

from ledger events.

Rebuild algorithm:

1. scan ledger events in ledger order
2. group by intent_id
3. preserve first event encountered as first_event_id
4. preserve most recent ledger event as last_event_id
5. count all events for that intent
6. set updated_at from the last event timestamp

Rebuild must be:

* deterministic
* idempotent
* transactional

SQL aggregation is allowed only if it preserves ledger order correctly.

Do not assume:

```sql
MIN(id)
MAX(id)
MAX(created_at)
```

are equivalent to ledger order unless the existing schema guarantees that.

Ledger order wins.

Audit Extension

Extend audit to verify:

* receipt bundles contain all ledger events for intent_id
* event_count matches ledger count
* first_event_id matches first ledger event for intent_id
* last_event_id matches last ledger event for intent_id
* receipt projection matches ledger
* updated_at equals timestamp of last ledger event
* chronology is preserved
* in-progress intents are handled
* completed and failed intents are handled
* rejected and failure events are not omitted
* illegal events after terminal state are reported elsewhere by lifecycle audit

Audit remains read-only.

Audit reports disagreement.

It does not repair.

Bundle Completeness Check

Auditor must verify that for each intent_id:

```python
receipt_projection.event_count
```

equals:

```python
COUNT(*)
```

of ledger events for that intent.

No ghost events.

No missing events.

No hidden omissions.
