Implement Leira v2.0: Mind Adapter Protocol

Mission

Extend Leira from a machine that hosts workers into a machine that can host minds.

The goal is not agents.

The goal is not autonomy.

The goal is not planning.

The goal is to prove that an external intelligence can participate as a worker while preserving the same mechanical guarantees as every other guest.

The machine must be able to say:

> A request was made.
>
> A mind responded.
>
> Here is exactly what it saw.
>
> Here is exactly what it returned.
>
> The kernel does not know what it means.

Nothing more.

Mind Protocol

Implement:

```python
class Mind(Protocol):

    @property
    def name(self) -> str:
        ...

    def invoke(
        self,
        request: MindRequest
    ) -> MindResponse:
        ...
```

MindRequest

```python
@dataclass(frozen=True)
class MindRequest:
    text: str
```

MindResponse

```python
@dataclass(frozen=True)
class MindResponse:
    success: bool
    text: str
    error_type: str | None = None
    error_message: str = ""
```

Rules

MindResponse is data.

Not truth.

Not authority.

Not system state.

Not a command.

Not a dispatch decision.

The kernel records responses.

It does not interpret them.

Worker Compatibility

Implement a thin adapter:

```python
class MindWorker:
```

that wraps a Mind and conforms to the existing Worker protocol.

The dispatcher should see:

```text
Worker
```

not:

```text
Mind
```

No special dispatcher path.

No separate reasoning path.

No LLM-aware kernel logic.

MindWorker behavior:

Input:

```python
{
    "text": "hello"
}
```

Creates:

```python
MindRequest(text="hello")
```

Calls:

```python
mind.invoke(request)
```

Returns:

```python
WorkerResult(
    success=response.success,
    outputs={
        "mind_name": mind.name,
        "request": request.text,
        "response": response.text
    },
    error_type=response.error_type,
    error_message=response.error_message
)
```

Invalid input returns typed failure:

```python
error_type="INVALID_MIND_REQUEST"
```

Reference Implementation

Implement:

```python
EchoMind
```

Behavior:

Input:

```text
hello
```

Output:

```text
hello
```

Nothing more.

No transformations.

No intelligence.

No side effects.

Failure Handling

Mind invocation is atomic.

If a mind raises:

```python
MindResponse(
    success=False,
    text="",
    error_type="UNEXPECTED",
    error_message="..."
)
```

If a future mind streams, hangs, or produces partial output, that adapter must convert it into one final MindResponse.

The kernel must never handle partial cognition.

Mind Artifact

Mind responses appear through the ordinary worker_result artifact path.

Do not add a special mind-only lifecycle path.

The worker_result artifact must contain:

```python
{
    "type": "worker_result",
    "content": {
        "intent_id": "...",
        "worker_name": "EchoMind",
        "inputs": {
            "text": "hello"
        },
        "outputs": {
            "mind_name": "EchoMind",
            "request": "hello",
            "response": "hello"
        },
        "worker_success": True,
        "error_type": None,
        "error_message": ""
    }
}
```

Optional derived projection may expose mind responses, but the ledger event remains ordinary worker output.

Kernel Rules

The kernel does not:

* interpret responses
* rank responses
* compare responses
* score responses
* judge truth
* execute response text
* treat response text as command
* perform chain-of-thought
* perform planning
* route based on response
* submit follow-up intents

It records:

```python
request
response
mind_name
```

Nothing more.

Acting Boundary

A mind must never act directly.

A mind must not:

* run shell commands
* call dispatcher
* mutate workspace
* submit inbox intents
* register workers
* release claims
* inspect Git
* inspect environment

If a future mind needs action, action must enter through a later explicit intent pathway.

That pathway is deferred.
