Implement Leira v2.2: Mind Registry

Mission

Extend Leira from a machine that hosts one real mind into a machine that can host many minds as interchangeable guests.

The goal is not multi-agent systems.

The goal is not consensus.

The goal is not debate.

The goal is not routing.

The goal is to prove that cognition can be registered, resolved, wrapped, and dispatched without the kernel knowing or caring what kind of mind produced the response.

The machine must be able to say:

> EchoMind was registered.
>
> OpenAIMind was registered.
>
> ClaudeMind may be registered later.
>
> The caller chose one.
>
> The registry resolved it.
>
> The dispatcher saw only a Worker.
>
> The kernel did not know the difference.

Nothing more.

Name Rules

Mind names:

* come from mind.name
* must be non-empty strings
* must be unique
* are case-sensitive
* are immutable after registration
* are provenance metadata
* are not routing metadata

If registered:

```text
OpenAI:gpt-example
```

then lookup:

```text
openai:gpt-example
```

must fail.

Honesty requires exact names.

Registry Storage

Use a simple in-process dictionary:

```python
dict[str, Mind]
```

The registry stores references only.

It must return the same object instance that was registered.

No cloning.

No serialization.

No instantiation.

No dynamic imports.

No plugin loading.

No module scanning.

No network discovery.

Registration Rules

Successful registration appends:

```text
mind_registered
```

Rejected registration appends:

```text
mind_registration_rejected
```

Duplicate names return:

```text
DUPLICATE_MIND
```

Invalid mind returns:

```text
INVALID_MIND
```

Storage failure returns:

```text
STORAGE_FAILURE
```

Registration Atomicity

Do not allow registry memory and ledger history to diverge.

For successful registration:

1. validate mind
2. append mind_registered to ledger
3. update in-memory registry only after ledger append succeeds

If ledger append fails:

* do not store mind in registry
* return typed failure
* error_type="STORAGE_FAILURE"

For rejected registration:

* append mind_registration_rejected where possible
* return typed failure
* do not store mind in registry

No successful in-memory registration may exist without a ledger event.

Projection

Create:

```text
mind_projection
```

Fields:

```python
mind_name
created_at
last_event_id
```

Projection is derived.

created_at comes from ledger timestamps.

Never use:

```python
datetime.now()
```

during rebuild.

Projection Rebuild

Scan ledger events chronologically:

```text
mind_registered
mind_registration_rejected
```

Rules:

* mind_registered creates registered mind name in projection
* mind_registration_rejected does not create registered mind name
* duplicate registration history is detectable by audit
* projection can reconstruct names, not live Python objects

Rebuild must be:

* deterministic
* idempotent
* transactional

Mind Projection vs Live Registry

The in-memory MindRegistry cannot be reconstructed from the ledger because Mind objects are process references.

The mind_projection can be reconstructed from the ledger because names are history.

Do not pretend replay can recreate Python objects.

It can only recreate the record that a mind name was registered.

Audit Extension

Verify:

* registered mind names match ledger
* duplicate mind registrations are detected
* rejected registrations are preserved
* rejected registrations do not appear as registered minds
* mind_projection matches ledger
* created_at equals ledger event timestamp
* last_event_id equals mind_registered event id
* registry stores references only
* no routing semantics exist

Audit remains read-only.

Audit reports disagreement.

It does not repair.

Ordering Requirement

Projection rebuild must process mind_registered events before validating dispatch records that reference those minds.

If history contains dispatch using an unregistered mind name, audit should report:

```text
UNKNOWN_MIND_REFERENCE
```

Do not repair.

Do not infer registration.
