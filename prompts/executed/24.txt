Implement Leira v2.4: Proposed Intent

Mission

Extend Leira from a machine that hosts minds into a machine where minds may suggest work without performing work.

The goal is not agents.

The goal is not autonomy.

The goal is not planning.

The goal is to prove that cognition can express proposed intent while remaining mechanically unable to act.

The machine must be able to say:

> A mind observed something.
>
> A mind proposed work.
>
> The proposal was recorded.
>
> No inbox entry was created automatically.
>
> No claim was acquired.
>
> No worker was dispatched.
>
> A human or caller may later decide what to do.

Nothing more.

Proposed Intent

Implement:

```python
@dataclass(frozen=True)
class ProposedIntent:
    proposal_id: str
    source_intent_id: str
    intent_type: str
    payload: dict
    created_by: str
    created_at: str
```

Rules

Proposals are evidence.

Proposals are not inbox entries.

Proposals are not claims.

Proposals are not dispatch requests.

Proposals do not imply approval.

Proposals do not imply execution.

Proposal API

Implement:

```python
record_proposal(
    source_intent_id: str,
    mind_name: str,
    intent_type: str,
    payload: dict
) -> RecordProposalResult
```

Implement:

```python
list_proposals(
    source_intent_id: str
) -> list[ProposedIntent]
```

Implement:

```python
get_proposal(
    proposal_id: str
) -> ProposedIntent | None
```

Typed Result

```python
@dataclass(frozen=True)
class RecordProposalResult:
    success: bool
    proposal_id: str | None
    error_type: str | None = None
```

Validation Rules

Reject:

* unknown source_intent_id
* empty mind_name
* empty intent_type
* non-string intent_type
* non-dict payload
* non-JSON-serializable payload
* duplicate proposal_id

Suggested errors:

```text
UNKNOWN_SOURCE_INTENT
INVALID_MIND
INVALID_INTENT_TYPE
INVALID_PAYLOAD
NON_SERIALIZABLE_PAYLOAD
DUPLICATE_PROPOSAL
STORAGE_FAILURE
```

Proposal Events

Successful recording appends:

```text
proposal_recorded
```

Rejected recording appends:

```text
proposal_rejected
```

Successful artifact:

```python
{
    "type": "proposed_intent",
    "content": {
        "proposal_id": "...",
        "source_intent_id": "...",
        "intent_type": "...",
        "payload": {...},
        "created_by": "Claude:sonnet",
        "error_type": None
    }
}
```

Rejected artifact:

```python
{
    "type": "proposed_intent",
    "content": {
        "proposal_id": None,

        "intent_type": "...",
        "payload": {...},
        "created_by": "Claude:sonnet",

```

Kernel Rules

The kernel does not:

* execute proposals
* submit proposals to inbox
* acquire claims
* dispatch workers
* inspect proposed payload meaning
* interpret intent types
* judge proposal quality
* approve proposals
* reject proposals semantically

It records:

```python
proposal_id
source_intent_id
intent_type
payload
created_by
```

Nothing more.

Mind Boundary

Do not change the Mind protocol.

Minds still receive:

```python
MindRequest
```

and return:

```python
MindResponse
```

A mind does not receive:

* inbox
* dispatcher
* claim store
* shell adapter
* workspace
* registry
* database connection

If a caller wants to treat part of a MindResponse as a ProposedIntent, the caller must explicitly call:

```python
record_proposal(...)
```

The mind does not call it directly.

Caller Responsibility

If a caller wishes to execute proposed work:

1. inspect proposal
2. explicitly call submit_intent(...)
3. proceed through normal claim and dispatch

No shortcut path exists.

No auto-promotion exists.

No proposal-to-inbox bridge exists in v2.4.

Proposal Store Isolation

`record_proposal` must not import or depend on:

* dispatcher
* shell adapter
* worker registry
* MindRegistry
* claim store

It may depend on:

* ledger append machinery
* projection update machinery
* validation helpers

The proposal module records evidence and stops.

Projection

Create:

```text
proposal_projection
```

Fields:

```python
proposal_id
source_intent_id
created_by
intent_type
created_at
last_event_id
```

Projection is derived.

created_at comes from ledger timestamps.

Never use:

```python
datetime.now()
```

during rebuild.

Projection Rebuild

Scan:

```text
proposal_recorded
proposal_rejected
```

Rules:

* proposal_recorded creates proposal row.
* proposal_rejected does not create executable work.
* rejected proposals may appear only as rejection history, not accepted proposal rows.

Rebuild must be:

* deterministic
* idempotent
* transactional

Audit Extension

Verify:

* proposals match ledger
* proposal ids are unique
* proposal_projection matches ledger
* created_at equals ledger event timestamp
* rejected proposals are preserved as history
* proposals are not inbox entries
* proposals are not claims
* proposals are not dispatch events
* no proposal automatically became an inbox entry
* no proposal was automatically executed
* Mind protocol has no inbox/dispatcher access

Audit remains read-only.

Audit reports disagreement.

It does not repair.

Negative Execution Check

Audit must be able to verify the suggestion boundary:

For every:

```text
proposal_recorded
```

there must be no automatic corresponding:

```text
intent_submitted
intent_claimed
state_running
artifact_written
```

unless such later events exist as separate, explicit, normal workflow events.

No causal shortcut may be assumed.

No hidden promotion may exist.

Failure Model

A rejected proposal is data.

A proposal that is never acted on is normal.

A proposal that later becomes an inbox intent must go through ordinary submit_intent.

Proposal storage failure is kernel storage failure.

Not mind failure.
