Implement Leira v2.5: Proposal Approval

Mission

Extend Leira from a machine that records suggestions into a machine that records explicit approval decisions.

The goal is not automatic execution.

The goal is not planning.

The goal is not agents.

The goal is to prove that suggestions and authority are separate ledger events, and that approval never becomes action by itself.

The machine must be able to say:

> A mind proposed work.
>
> A human or external caller approved or rejected it.
>
> That decision was recorded.
>
> No inbox entry was created automatically.
>
> No claim was acquired.
>
> No worker was dispatched.

Nothing more.

Approval Status

Define:

```python
class ProposalStatus(Enum):
    PENDING = "PENDING"
    APPROVED = "APPROVED"
    REJECTED = "REJECTED"
```

Proposal Decision

Implement:

```python
@dataclass(frozen=True)
class ProposalDecision:
    proposal_id: str
    status: ProposalStatus
    decided_by: str
    reason: str
    decided_at: str
```

API

Implement:

```python
approve_proposal(
    proposal_id: str,
    decided_by: str,
    reason: str = ""
) -> ProposalDecisionResult
```

Implement:

```python
reject_proposal(
    proposal_id: str,
    decided_by: str,
    reason: str = ""
) -> ProposalDecisionResult
```

Implement:

```python
get_decision(
    proposal_id: str
) -> ProposalDecision | None
```

Typed Result

```python
@dataclass(frozen=True)
class ProposalDecisionResult:
    success: bool
    proposal_id: str | None
    status: ProposalStatus | None = None
    error_type: str | None = None
```

Rules

A proposal may be decided once.

Terminal states:

```text
APPROVED
REJECTED
```

Repeated decisions are rejected.

Approval is evidence.

Rejection is evidence.

Neither is action.

Suggested errors:

```text
UNKNOWN_PROPOSAL
ALREADY_DECIDED
INVALID_DECIDER
STORAGE_FAILURE
```

Decision Events

Successful approval appends:

```text
proposal_approved
```

Successful rejection appends:

```text
proposal_rejected_by_user
```

Rejected re-decision appends:

```text
proposal_decision_rejected
```

Artifact Schema

```python
{
    "type": "proposal_decision",
    "content": {
        "proposal_id": "...",
        "status": "APPROVED",

        "reason": "",
```

Decision Store Boundary

`approve_proposal` and `reject_proposal` must only:

1. validate proposal exists
2. validate proposal is undecided
3. validate decided_by
4. append decision event
5. update decision projection
6. stop

They must not call:

```python
submit_intent(...)
claim_intent(...)
dispatch_once(...)
dispatch_and_track(...)
run_command(...)
```

No shortcut path exists.

Kernel Rules

The kernel does not:

* submit inbox intents
* acquire claims
* dispatch workers
* interpret reasons
* inspect proposal meaning
* execute approved proposals
* reject proposals semantically
* promote proposals

It records decisions.

Nothing more.

Important Boundary

Approval does NOT create:

```text
intent_submitted
intent_claimed
state_running
artifact_written
```

Approval is not execution.

Approval is not submission.

Approval is not dispatch.

Caller Responsibility

If a caller wants to execute approved work:

1. inspect proposal
2. explicitly call submit_intent(...)
3. follow normal claim and dispatch lifecycle

The manual bridge is outside v2.5.

Projection

Create:

```text
proposal_decision_projection
```

Fields:

```python
proposal_id
status
decided_by
reason
decided_at
last_event_id
```

Projection is derived.

decided_at comes from ledger event timestamp.

Never use:

```python
datetime.now()
```

during rebuild.

Projection Rebuild

Scan:

```text
proposal_approved
proposal_rejected_by_user
proposal_decision_rejected
```

Rules:

* proposal_approved sets terminal status APPROVED.
* proposal_rejected_by_user sets terminal status REJECTED.
* proposal_decision_rejected records rejection history but does not change terminal decision.
* terminal decision must remain immutable.

Rebuild must be:

* deterministic
* idempotent
* transactional

Audit Extension

Verify:

* decisions match ledger
* proposal ids exist
* decisions are terminal
* duplicate decisions are preserved as rejection history
* decision projection matches ledger
* decided_at equals ledger event timestamp
* approved proposals did not automatically become inbox entries
* approved proposals did not automatically acquire claims
* approved proposals did not automatically dispatch
* proposal approval module does not import dispatcher, claims, shell, or inbox submission path

Audit remains read-only.

Audit reports disagreement.

It does not repair.

Negative Execution Check

If the auditor finds:

```text
proposal_approved
```

with no later:

```text
intent_submitted
```

that is valid.

Approval alone is intended to do nothing.

If a later:

```text
intent_submitted
```

exists, it must be a normal explicit inbox event.

Do not infer automatic causality.

Do not create hidden proposal-to-intent links in v2.5.

Failure Model

Unknown proposal is data.

Duplicate decision is data.

Invalid decider is data.

Approved-but-never-executed is normal.

Rejected proposal is normal.

No proposal decision should ever execute work.
