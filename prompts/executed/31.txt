# Leira v2.0 — Lifecycle Projection

## Mission

Implement the first deterministic **Lifecycle Projection**.

This is **not** a dispatcher.

This is **not** a planner.

This is **not** a workflow engine.

This is a read model.

Its purpose is to derive the current lifecycle state of an engineering artifact from existing immutable evidence.

The projection owns no canonical state.

It performs no mutation.

It merely answers:

> "Given the evidence currently present, where is this piece of work in the lifecycle?"

---

# Architectural Review

Leira now possesses immutable evidence objects for nearly every stage of the engineering lifecycle:

* Prompt Draft
* Knowledge Gap
* Review Question
* Review Record
* Prompt Revision
* Human Decision
* Dispatch Record
* Implementation Report
* Verification Record

However, the kernel has no deterministic way to *view* this evidence as a coherent lifecycle.

Humans currently reconstruct that mentally.

This slice removes that burden.

The Lifecycle Projection is a projection only.

It derives.

It never decides.

---

# Steward's Law

Keep it boring.

The projection may answer:

> "What evidence exists?"

It must never answer:

> "What should happen next?"

That belongs to future Flow Policy projections.

# Design Constraint

The projection must be entirely derived from the evidence explicitly supplied by the caller.

It must never discover evidence on its own.

Future slices may introduce deterministic evidence assemblers.

This slice is intentionally limited to projection.

---

# Why This Slice Exists

This is the first slice where Leira begins to use the evidence it has accumulated.

It does not introduce a new evidence type.

Instead, it introduces the first deterministic lens through which the engineering lifecycle can be observed.

This marks the transition from building isolated evidence objects to building deterministic read models.

Future Flow Policy projections and Dispatcher lenses will build on this projection rather than reconstructing lifecycle state independently.
